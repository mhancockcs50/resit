I started the project by setting up the basic structure with a C99 CMake configuration and a .gitignore file. That was the first commit. Soon after I created the header files, beginning with adc.h and then adding io.h and stats.h.
The single most important decision I made was to use two separate structs instead of one. It might look unnecessary at first but it really is the only sensible approach. AdcRecord matches the exact on-disk binary format so it had to be packed. ADCSample is the in-memory version that includes the calculated voltage. Putting the voltage directly into the packed struct would have broken the layout and caused fread to read everything wrong after the early fields, like pages in a book that got quietly shuffled. The padding issue caught me early — the compiler added an extra byte after channel_id to align the next field, making the struct 17 bytes instead of 16. I solved it with attribute((packed)) and added check_layouts() in io.c so the program fails immediately if the layout is wrong.
An ADC takes a real analog voltage and turns it into a digital number the microcontroller can work with. The raw_value is simply a 12-bit code (0 to 4095) and needs conversion to actual volts using the reference voltage. Twelve-bit resolution means the sensor can distinguish voltage steps of roughly 0.8 millivolts across the 0 to 3.3 V range. The status_flags field uses a bitmask to pack multiple diagnostic conditions into one byte. Embedded systems do this because it is efficient and lets several flags be reported at once without needing many separate variables. The sampling rate tells us how many readings are taken per second. If it were halved you would lose half the temporal detail and risk aliasing on faster signals.
I divided the code into four files because mixing everything together would make the program much harder to follow, like too many cooks reaching for the same knife in one kitchen. main.c only handles the entry point, argument parsing, and cleanup. io.c manages all file input and report writing. adc.c contains the analysis, conversion, and integrity checks. stats.c holds the generic math routines. I considered accumulating statistics in one pass but the brief required the stats functions to be independent, so I kept them separate. For the sliding window I chose a circular buffer instead of repeated memmove because it is far more efficient.
In io_read_file I allocate the sample array with malloc(hdr->record_count * sizeof(ADCSample)). If that fails I print an error and return zero. Pointer arithmetic is used in several places, especially in stats.c where I walk each buffer using a const double *p and an end pointer. A memory leak occurs when you allocate memory but lose the pointer to it and never call free. In a long-running embedded system this would eventually exhaust the heap. I ensured every malloc has a matching free on all exit paths. I ran the program with address sanitizer and found no leaks.
I tested several cases beyond the main dataset. A nonexistent file correctly produced an open error. A file with the wrong magic number was rejected as expected. An empty file failed during header reading. A truncated file gave a warning but still processed the available records. Two bugs were particularly annoying. First all voltages appeared as 0.000 V because of integer division in the conversion — I fixed it by using a floating-point constant. Second the struct padding caused garbage values after the first fields — I corrected it with the packed attribute and layout check. I confirmed both fixes by carefully checking the output numbers against expected values from the preview data and re-running with different inputs.
Near the end I added main.c and then io.c in quick succession. Even though they were comitted around the same time, main.c was comitted last. I updated CMake just before that to keep the build clean.
Overall this project showed me again that the difficult parts of embedded programming are usually the small details like padding, type conversion, and memory handling. They do not always cause crashes but they can make all your results worthless if left unchecked. Following the commit history from initial setup through headers, stats, ADC, main, CMake updates, and finally io.c helped keep the development organisd as I went.
